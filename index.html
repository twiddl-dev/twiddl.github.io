> Technical Notes

  • Nodes run on your machines as Docker containers with minimal footprint.
  • Specified folders are monitored for any changes in files structure.
  • You can join and query any files across all nodes using Twiddl web interface.
  • SQL or Drag and Drop interface — friendly for any level of expertise.
  • Each node processes its own batch of data locally and transmits only the results.
  • Twiddl server aggregates results from nodes and presents them in one place.
  • Supported formats: XLSX, CSV, JSON, Parquet, XML, AVRO and more.
  • The node is lightweight enough to run on IoT devices for live monitoring.
  • Deploy Twiddl server on your local network, or use our web service.

04 UPDATES

Proof we’re building: real milestones over time

Mar 01, 2026 · Update 06

Preparation Stage: Making Twiddl Demo-Ready for Everyone

This is the “make it real for leads” phase: not new hype features — just a clean, repeatable experience that anyone can try without scheduling around my laptop.

The vision is simple: one public central node, a couple of connected local nodes, and a synthetic dataset that demonstrates the distributed query flow end-to-end — reliably.

  • Deploy central in the cloud behind HTTPS/WSS so it’s reachable 24/7
  • Connect internal local nodes that phone home and stay online
  • Ship a stable “happy path”: a few queries that always show the point
  • Turn demos into conversations: walkthroughs, Q&A, and pilot readiness

If you’re evaluating Twiddl, this is where you’ll be able to test it like a product — not a prototype.

Oct 08, 2025 · Update 01

The First Obsession: “Why can’t I just query all of this?”

I kept running into the same friction: CSVs, JSON exports, XLSX files, and logs — and every “analysis” attempt turned into pipeline work. I didn’t want a pipeline. I wanted a question.

So I started with a very practical experiment: can I treat mixed file formats as a single queryable surface using an embedded SQL engine? DuckDB became the workhorse because it’s fast, local-first, and surprisingly good at reading “real world” files.

That early stage was messy in the best way: schema drift, messy headers, half-broken JSON, inconsistent timestamps. But it clarified the direction: Twiddl should be SQL-first and file-native, and it should do the boring work of making heterogeneous data feel coherent.

What surprised me most wasn’t the SQL part—it was how much time gets wasted on “preparing” data that’s already valuable. Twiddl started as a refusal to keep writing one-off scripts just to answer one question, then throwing them away.

  • DuckDB is an excellent local engine for “data-in-files,” but the moment files are scattered across machines, the problem becomes distribution, not SQL.
  • The “one format to rule them all” dream is less important than one interface to ask questions.
  • The real enemy is the delay between “I have data” and “I can learn something from it.”
Oct 23, 2025 · Update 02

The Remote File That Forced a New Architecture

The turning point was simple: the files I wanted to test were on another machine… in another location. I could wait — or accept that “data lives where it lives.”

The deeper problem: copying is not only slow, it’s risky. The moment you start moving files around “just for analysis,” you create version confusion, accidental leaks, and permission questions you didn’t mean to trigger.

Instead of building yet another “centralize everything first” system, I leaned into the constraint:

  • Keep data where it is
  • Run queries where the data lives
  • Coordinate and aggregate results centrally

That’s when the concept of local nodes and a central node solidified — not as a buzzword, just as the simplest way to make a distributed reality usable.

  • A philosophy that later shaped security choices: authenticate nodes, validate messages, and avoid “just upload your data” as the default.
Nov 12, 2025 · Update 03

First Working Prototype: Two Machines, One End-to-End Query

I set a goal that felt small but was actually huge: make one query travel to another machine, execute locally, and come back with results.

The first version was intentionally constrained: two machines, same local network, same OS assumptions, and a narrow set of file types. But the entire loop mattered: local metadata, heartbeats, remote dispatch, and results returning in one cycle.

  • A local node watches a directory of files and infers metadata/schemas using DuckDB.
  • It maintains a persistent WebSocket client connection to the central node.
  • The central node accepts metadata + heartbeats, then dispatches query requests.
  • The local node executes subqueries and streams rows back in chunks.

What made this milestone feel “real” was that it wasn’t a toy RPC. The local node had to behave like a worker you can rely on; the central node had to act like a coordinator that knows where files live and routes queries correctly.

Dec 09, 2025 · Update 04

The “Cafe Demo” Moment: Remote Central + Local Laptop Node

After the LAN prototype worked, I took it outside the lab and ran a live demo on public Wi‑Fi. That’s where distributed systems become honest.

  • Networking quirks are not theoretical.
  • You need clear “online/offline/stale” state.
  • Reconnect behavior has to be automatic (because nobody wants to babysit a node).

It also forced the product story to sharpen. In a cafe, you don’t have time to explain a distributed architecture diagram — you have to show something simple: files on a machine, a central view, a query, and results.

Feb 25, 2026 · Update 05

Launcher, macOS Testing, and a Security Baseline (Current State)

This stage was about trust and usability: easier startup flow, cross-environment validation, and a clear security baseline for demos and pilots.

  • Launcher and run flow: the local launcher starts the local node container with a mounted watch directory.
  • Cross-environment testing (Mac): validated that launcher/run assumptions hold on macOS setups.
  • Security + message validation: central HTTP uses basic auth; local↔central WS messages use API key + HMAC signature and are validated server-side.

On the data path side, local nodes stream query results back in compressed chunks, and the central node aggregates and paginates results (with CSV export). This is the point where the runtime is stable enough to put in front of leads.

One detail I care about: this isn’t “security theater.” It’s a pragmatic line in the sand for demos and pilots. Nodes must prove who they are, and the central node should be able to say “no” by default.
